## 课时设计原则

### 可视化要求

每课至少包含以下之一:
- ASCII art 架构/流程图
- 搜索过程的逐步可视化（如 HNSW 的多层搜索过程）
- 对比表格
- 内存布局图（字节偏移标注）
- recall-QPS 曲线的示意图

### 源码准确性要求

- **禁止凭印象编写代码片段。** 写课时前必须用 Read 工具读取实际源码，确认:
  - 成员变量名和类型
  - 函数签名和参数列表
  - 行号（标注后再用 Read 验证）
- **伪代码与真实代码明确区分**: 伪代码标注"(简化)"，真实代码标注文件路径和行号
- **数值计算写完整公式**: 涉及字节数、参数范围时，从源码找到计算公式，代入具体值

### 向量领域特别注意

- **距离计算必须讲 SIMD**: 距离计算是向量引擎最热的代码路径，不能只讲数学公式
- **量化公式必须配直觉**: 每个 LaTeX 公式前先用一句话说清"到底在干什么"
- **内存布局要画字节图**: 向量引擎的性能很大程度取决于内存布局，要画出每个字段的偏移
- **参数影响要有曲线**: M、ef、nprobe 等参数的影响用 recall-QPS 关系说明

### 封装层/胶水层课时指南

向量库中常见的胶水层:
- Index 基类虚函数封装（如 faiss::Index）
- Python/Java binding 层
- 参数解析 + 索引工厂（如 faiss index_factory）

应聚焦"为什么需要这层"，而非罗列 API:
1. **用对照表展示 gap**: 算法层提供的 vs 用户需要的
2. **关键设计模式**: 异常边界转换、线程安全策略、状态管理
3. **非平凡逻辑**: 不是简单转发的部分
4. 如果信息量不足以独立成课，合并到相邻课时

### 深度均匀性检查

生成全部课时后，回顾检查:
- 相邻课时的信息密度是否差距过大（不应出现某课 400 行、某课 100 行）
- 对比类内容（如 A vs B）双方是否达到相近的讲解深度
- 如果某个主题确实只需概览，在 README 中明确标注为"概览"级别
