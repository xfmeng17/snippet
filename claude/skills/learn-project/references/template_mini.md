# 模式 C 模板：手写 Mini 版

## TASK.md 模板

```markdown
# {MiniProjectName} - {一句话定位}

## 目标
通过从零手写 {原项目} 核心流程的方式，深入理解 {主题}。
搞清楚三件事：
1. **How** —— 主干流程是怎么设计、怎么实现的
2. **Why** —— 为什么这样设计？解决了什么问题？有哪些取舍？
3. **Simplify** —— Mini 版做了哪些简化？为什么可以简化？

## 参考源码
- 原项目: `{path}`
- 关键模块速查见 `docs/architecture.md`

## 技术约束
- 语言: ...
- 构建: ...
- 测试: ...
- 代码风格: ...

## 核心流程
用 ASCII art 画出要实现的主干流程

## 子任务拆解

### Phase 1: {阶段名}
| # | 子任务 | 说明 | 状态 |
|---|--------|------|------|
| 01 | ... | ... | TODO |

### Phase 2: {阶段名}
...
```

## Task PRD 模板 (tasks/taskNN_xxx.md)

```markdown
# Task NN: {任务名}

## 目标
要构建什么（一句话）

## 为什么（Why）
- 在原项目中，这个模块解决什么问题
- 为什么这样设计（原项目的设计理念）
- Mini 版做了哪些简化，为什么可以简化

## 需求
具体要实现什么：
- 文件: `{project}/{module}/{file}.h/cpp`
- 类/函数: ...
- 接口: ...

## 设计要求
- 架构约束
- 与原项目的对应关系

## 验收标准
- [ ] 编译通过
- [ ] 单测通过: tests/{module}/xxx_test.cpp
- [ ] 功能验证: ...

## 参考
- 原项目对应源码: `{path}`
- 相关文档: ...
```

## architecture.md 模板

```markdown
# {MiniProject} 架构 —— 与 {原项目} 的对照

## 模块对照

| Mini 版模块 | 原项目对应 | 简化说明 |
|-------------|-----------|---------|
| {module1} | {original_path1} | 去掉了 XX，只保留 YY |
| ... | ... | ... |

## 数据流
ASCII art 画出 Mini 版的数据流

## 目录结构
树形结构 + 每个目录的简要说明
```

## .claude/CLAUDE.md 模板

```markdown
# {MiniProject} 开发规则

## 核心原则
- 每个模块必须写清楚 "Why"
- 先写测试，再写实现
- 完成一个 task 就提交一次

## 代码风格
- ...

## 测试要求
- 每个函数对应单测
- 测试输出要有可读的日志

## 提交规范
- feat/fix/docs + task 编号
```
